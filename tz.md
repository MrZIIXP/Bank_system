
# Техническое задание

## Название проекта

Backend API для банковской/платёжной системы с OTP-авторизацией, аккаунтами, картами, переводами, кредитом, депозитом, историей операций, чёрным списком и админ-доступом.

---

## 1. Цель проекта

Разработать backend API, в котором пользователь может:

* пройти авторизацию по номеру телефона через One Time Password;
* создать аккаунт после подтверждения OTP;
* добавить банковскую карту;
* проверить существование аккаунта по телефону и карты по номеру;
* выполнять переводы между аккаунтами;
* выполнять внутренние переводы по номеру телефона или номеру карты;
* получать кредит для карты;
* делать депозит для карты;
* просматривать историю операций с фильтрацией;
* работать с black list для аккаунтов и карт;
* предоставить администратору доступ ко всей информации и управлению всеми данными.

---

## 2. Основные сущности системы

## 2.1. Account

Сущность аккаунта создаётся после успешной верификации OTP.

Поля:

* `user` — пользователь, берётся из request;
* `fname` — имя;
* `lname` — фамилия;
* `passport_id` — ID паспорта;
* `balance` — баланс аккаунта, по умолчанию `0`.

---

## 2.2. Card

Карта привязана к аккаунту пользователя.

Поля:

* `account` — аккаунт, берётся по пользователю из request;
* `card_id` — 16-значный номер карты;
* `balance` — баланс карты, по умолчанию `0`;
* `cart_name` — тип карты, choices:

  * `visa`
  * `credit`
  * `master`
  * `simple`
* `cvv` — случайное 3-значное число;
* `created_at` — дата создания, автоматически текущее время;
* `expair` — срок действия карты, по умолчанию `5 лет` от даты создания.

---

## 2.3. Transaction

Перевод между аккаунтами.

Поля:

* `type` — тип перевода, choices:

  * `phone_num`
  * `card`
* `sender` — аккаунт отправителя;
* `reciver` — аккаунт получателя;
* `amount` — сумма перевода;
* `created_at` — дата и время создания;
* `cuur_balance_sender` — текущий баланс отправителя после/во время операции;
* `cuur_balance_reciver` — текущий баланс получателя после/во время операции;
* `description` — описание перевода;
* `status` — статус операции.

---

## 2.4. Transaction Inside

Внутренний перевод по номеру телефона или номеру карты.

Поля:

* `type` — тип перевода, choices:

  * `phone_num`
  * `card`
* `sender` — отправитель, это либо `phone_num`, либо `card`;
* `reciver` — получатель, это либо `phone_num`, либо `card`;
* `amount` — сумма перевода;
* `created_at` — дата и время создания;
* `cuur_balance_sender` — текущий баланс отправителя;
* `cuur_balance_reciver` — текущий баланс получателя;
* `description` — описание перевода;
* `status` — статус операции.

---

## 2.5. Get Credit

Операция получения кредита для карты.

Поля:

* `card_id` — номер карты;
* `amount` — сумма кредита;
* `created_at` — дата создания;
* `procent` — процент кредита;
* `status` — статус.

Дополнительное условие:

* перед оформлением кредита необходимо проверить, что карта имеет тип `credit`.

---

## 2.6. Put Deposit

Операция депозита для карты.

Поля:

* `card_id` — номер карты;
* `amount` — сумма депозита;
* `created_at` — дата создания;
* `procent` — процент депозита;
* `status` — статус.

---

## 2.7. Black List для аккаунта

Поля:

* `account` — аккаунт;
* `created_at` — дата создания;
* `description` — причина/описание.

---

## 2.8. Black List для карты

Поля:

* `card` — карта;
* `created_at` — дата создания;
* `description` — причина/описание.

---

## 3. API-модули

# 3.1. Auth

## 3.1.1. `auth`

Назначение:

* отправка OTP на указанный номер телефона.

Входные данные:

* `phone_num`

Логика:

* пользователь отправляет номер телефона;
* система отправляет One Time Password на этот номер.

---

## 3.1.2. `verify`

Назначение:

* подтверждение OTP и создание аккаунта.

Входные данные:

* данные для проверки OTP;
* `fname`
* `lname`
* `passport_id`

Логика:

* после успешной проверки OTP создаётся `account`;
* поле `user` берётся из request;
* `balance` создаётся со значением `0`.

Результат:

* созданный аккаунт пользователя.

---

# 3.2. Card

## 3.2.1. `add_card`

Назначение:

* создание карты для аккаунта пользователя.



Входные данные:

* `16_digit(card_id)`
* `cart_name`

Логика:

* `account` определяется из пользователя в request;
* карта создаётся с балансом `0`;
* `cvv` генерируется случайно как 3-значное число;
* `created_at` заполняется автоматически;
* `expair` устанавливается на 5 лет вперёд.

Результат:

* созданная карта.

---

## 3.2.2. `check_if_account_exsits`

Назначение:

* проверка существования аккаунта и карты.

Входные данные:

* `phone_num`
* `16_digit(card_id)`

Логика:

* система проверяет, существует ли аккаунт по номеру телефона;
* система проверяет, существует ли карта по номеру карты.

Результат:

* информация о наличии или отсутствии аккаунта и карты.

---

# 3.3. Transactions

## 3.3.1. `transaction`

Назначение:

* перевод между аккаунтами.

Входные данные:

* `type`
* `sender`
* `reciver`
* `amount`
* `description`

Логика:

* создаётся запись перевода;
* сохраняются отправитель и получатель;
* сохраняется сумма;
* сохраняется текущее время;
* сохраняются текущие балансы отправителя и получателя;
* сохраняется описание;
* сохраняется статус операции.

Результат:

* информация о переводе.

---

## 3.3.2. `transaction_inside`

Назначение:

* внутренний перевод по номеру телефона или по карте.

Входные данные:

* `type`
* `sender` — телефон или карта
* `reciver` — телефон или карта
* `amount`
* `description`

Логика:

* если выбран тип `phone_num`, то sender и reciver работают как номера телефонов;
* если выбран тип `card`, то sender и reciver работают как номера карт;
* создаётся запись операции;
* фиксируются балансы сторон;
* сохраняются дата, описание и статус.

Результат:

* информация о внутреннем переводе.

---

# 3.4. Credit and Deposit

## 3.4.1. `get_creadit`

Назначение:

* оформление кредита на карту.

Входные данные:

* `card_id`
* `amount`
* `procent`

Логика:

* перед созданием операции проверяется тип карты;
* кредит можно получить только если карта является `credit`;
* после этого создаётся запись кредита со статусом и датой.

Результат:

* информация о кредите.

---

## 3.4.2. `put_deposit`

Назначение:

* оформление депозита на карту.

Входные данные:

* `card_id`
* `amount`
* `procent`

Логика:

* создаётся запись депозита;
* сохраняются сумма, процент, дата и статус.

Результат:

* информация о депозите.

---

# 3.5. History

## 3.5.1. `history`

Назначение:

* получение истории транзакций.

Поддерживаемые фильтры:

* `card`
* `income and pays`
* `inside`
* `time`

Логика:

* пользователь получает список операций;
* список можно фильтровать по карте;
* можно разделять входящие и исходящие платежи;
* можно смотреть внутренние переводы;
* можно фильтровать по времени.

Результат:

* `transaction_history` по заданным фильтрам.

---

# 3.6. Black List

## 3.6.1. Black list для аккаунта

Назначение:

* добавление аккаунта в чёрный список.

Данные:

* `account`
* `created_at`
* `description`

Логика:

* создаётся запись black list для аккаунта с датой и причиной.

---

## 3.6.2. Black list для карты

Назначение:

* добавление карты в чёрный список.

Данные:

* `card`
* `created_at`
* `description`

Логика:

* создаётся запись black list для карты с датой и причиной.

---

# 3.7. Admin

## 3.7.1. `admin`

Назначение:

* администратор читает всю информацию системы и управляет всеми данными.

Возможности:

* читать все аккаунты;
* читать все карты;
* читать все транзакции;
* читать все внутренние транзакции;
* читать все кредиты;
* читать все депозиты;
* читать все записи black list;
* управлять всеми перечисленными данными.

## 4. Бизнес-правила

### 4.1. Авторизация

* авторизация выполняется через `phone_num`;
* используется `One Time Password`.

### 4.2. Создание аккаунта

* аккаунт создаётся только после `verify`;
* баланс аккаунта по умолчанию равен `0`.

### 4.3. Создание карты

* карта создаётся для аккаунта пользователя из request;
* номер карты должен быть 16-значным;
* `cvv` должен быть случайным 3-значным числом;
* дата создания ставится автоматически;
* срок действия по умолчанию — 5 лет.

### 4.4. Переводы


* у каждой транзакции должен быть тип;
* у каждой транзакции должен быть отправитель и получатель;
* у каждой транзакции должна быть сумма;
* должен сохраняться статус;
* должны сохраняться текущие балансы отправителя и получателя;
* должно сохраняться описание;
* должна сохраняться дата создания.

### 4.5. Кредит

* кредит доступен только для карты типа `credit`.

### 4.6. История

* история операций должна поддерживать фильтрацию по:

  * карте;
  * входящим и исходящим операциям;
  * внутренним переводам;
  * времени.

### 4.7. Black list

* black list ведётся отдельно для аккаунтов и отдельно для карт;
* в каждой записи должна быть дата и описание причины.
